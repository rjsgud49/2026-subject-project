| 항목 | 내용 |
|------|------|
| **문서 버전** | 1.0 |
| **대상 단계** | P1 (고객·학생 기능) |

본 문서는 SW 분석·설계·구현·유지 단계에서 검토, 평가, 승인의 기준이 되는 문서이며, 기능적·비기능적 요구사항으로 구성된다.

# 1. 개요

## 1.1 범위

| 구분 | 범위 기술 |
|------|-----------|
| **시스템 범위** | 면접(인터뷰 준비) 온라인 강의 플랫폼의 **고객(학생) 관점 기능** |
| **포함** | 강의 검색, 수강신청(장바구니·간이 결제), 내 수업 조회, 수업 듣기(영상 재생·진도), 강의별 Q&A(질문·답변) |
| **제외** | 관리자·강사 기능, 실 결제 연동, 역할 기반 인증 세분화, 모니터링·로깅·알림 (→ P2·P3) |
| **산출물** | 백엔드 REST API, 프론트엔드 웹 UI, RDB 기반 데이터 저장 |

요구사항 ID는 **FR**(기능 요구사항), **EIR**(외부 인터페이스 요구사항), **NFR**(비기능 요구사항), **AC**(인수 조건)로 구분하며, 변경 시 추적 가능하도록 유지한다.

---

## 1.2 목적

- 본 요구사항 명세서의 **작성 목적**은 다음과 같다.
  - P1 단계에서 구현·검증할 **기능적·비기능적 요구사항**을 명확히 정의한다.
  - 분석·설계·구현·유지보수 시 **검토·평가·승인**의 기준 문서로 사용한다.
  - **인수 시험**의 기준(기능 시험·성능 시험 항목)을 제시한다.
  - 요구사항 간 **추적성**을 확보하여 변경 영향도 분석을 용이하게 한다.

## 1.3 시스템 개요

| 항목 | 내용 |
|------|------|
| **시스템명** | 면접 인강 플랫폼 (P1) |
| **요약** | 취업·이직 준비생(학생)이 면접 강의를 **검색**하고, **수강신청**한 뒤, **내 수업**에서 **영상을 시청**하며 **질문(Q&A)**을 할 수 있는 웹 기반 서비스이다. |
| **주요 사용자** | 학생(수강생) |
| **주요 흐름** | 검색 → 강의 상세 → 장바구니/수강신청 → 내 수업 → 수업 듣기(진도 저장) / 강의별 Q&A |

# 2. 기능적 요구사항

## 2.1 기능 요구사항 (기본 동작)

**특이성**: 중요도(M: 필수, S: 선택), 난이도(상/중/하), 변경 가능성(상/중/하)을 표기한다.

| NFR-D04 | **인덱스** | 목록 검색(카테고리, 난이도, 가격, 공개 여부), 내 수업(user_id), Q&A(course_id) 조회에 필요한 인덱스를 권장한다. | erd.md |

## 3.4 소프트웨어 시스템 속성

# 5. 요구사항 작성 원칙 적용 (IEEE 29148)

| 원칙 | 적용 내용 |
|------|-----------|
| **완전성** | P1 범위 내 필요한 기능·비기능 요구사항을 누락 없이 기술하였으며, 범위 밖(P2·P3)은 1.1·1.4에서 명시하였다. | |
| **정확성** | 각 요구사항은 "~해야 한다" 형태로 기술하여 해석의 여지를 줄였다. | |
| **명확성** | 용어(진도율, 수료, 이어보기, 간이 결제 등)를 1.3·2.1에서 정의·사용하였다. | |
| **일관성** | FR/EIR/NFR/AC ID 체계를 통일하였고, 기획서·erd·api.yml과 용어·범위를 맞추었다. | |
| **특이성** | 2.1에서 중요도(M/S), 난이도, 변경가능성, 검증 방법을 표기하였다. | |
| **검증가능성** | 각 기능 요구사항에 "검증 방법"을, 비기능에 "검증 방법/기준"을, 4장에 인수 조건을 제시하였다. | |
| **수정용이성** | ID 부여로 항목별 수정이 가능하며, 변경 시 1.4 GC-05에 따라 연관 문서와의 일관성 유지가 필요하다. | |
| **추적성** | 요구사항 ID(FR-xxx, EIR-xxx, NFR-xxx, AC-xxx)로 설계·구현·테스트에서 참조 가능하다. | |
| **이해가능성** | 개요→기능→비기능→인수 순으로 구성하고, 표와 목차로 가독성을 높였다. | |
